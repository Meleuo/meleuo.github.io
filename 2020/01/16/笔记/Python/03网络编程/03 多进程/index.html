进程理论

什么是进程

正在运行的程序称之为进程

进程是资源分配的基本单位，也是CPU调度的基本单位

进程的并发和并行

主观：并行和并发都是同时执行的2个进程

宏观：在CPU层面，并行是指N个进程被同时由N个CPU核心同一个时间点运行，但是并发则是由单个CPU核心依次再同一个时间段中运行^CPU分时系统。

进程的三个状态

就绪：任务已经递交给系统，并且获得除了CPU外的所有资源，一旦获得CPU则会立马执行

执行/运行： 当进程获得CPU^时间片后，程序就会在处理机上运行。

阻塞^非阻塞：执行的过程中由于某个事件无法继续执行，便会放弃CPU，然后等待阻塞结束，引起程序的阻塞有很多原因，比如等待IO完成，等待信号等。

进程有一个特殊的状态就是挂起，其行为就是由于某种原因，进程主动放弃了CPU，进程无法运行，被提出内存。

异步和同步

同步：并发中有 A主任务和子任务A1，A主任务启动A1，然后等待A1执行结束后，在执行A剩下的代码，这叫做同步，同步的特点就是可靠性高，要么全部成功，要么全部失败。

异步：并发中有 A主任务和子任务A1，A主任务启动A1，然后立马执行剩下的代码，然后等待A1执行完毕代码结束，异步的特点就是相对于同步要快，使用于主任务和子任务的数据没有任何逻辑关系的应用场景，即使A1失败了也不影响A。

multiprocessing 多线程

Python 中提供了一个 multiprocessing 模块，用来创建多线程[^注释]

严格来说 multiprocessing 并不是一个模块，而是一个管理进程的包，这个包里包含了几乎所有和进程有关的模块，但是为了便于理解，我下文中还是以一个模块形容 multiprocessing

[^注释]: multiprocessing 创建的多线程都是用的 并发 模式，并不支持 并行,

CPU的运行速度，熟快熟慢是说不准的，代码运行顺序一般是从上至下，但是解释器解析代码然后交给CPU运行，CPU运行这个代码也是需要时间的，解析器把信号发给系统后就去执行剩下的代码了，此时系统执行这个任务会有一个短暂的延迟情况。如果下一个任务和上一个相关，就有可能出现无法预料的错误（问题）

银行取钱

定义2个方法，一个取钱，一个存钱，都运行100次，理论上来说结果应该是不变的的，但是因为2个进程在同时运行（无所谓并发并行），在我们看来是同时运行的，但是宏观来说在CPU层面进程的运行是有先后顺序的，这个顺序就是CPU给进程分配的时间片决定的。

假设我的钱是 money=1，func1要存1块钱就是money + 1，func2要取一块钱就是menoy - 1, func1在对money进行 +1 时，执行到临时变量赋值[^临时变量]给money时，时间片消耗完了，需要等待CPU再次分配时间片给func1，他才能继续运行，但是func2此时开始运行，开始取钱了，成功执行了 money = money - 1，此时的money为 0， 因为func1已经执行到了临时变量赋值[^临时变量]赋值的操作了，所有就直接把运算完的临时变量赋值给了 money，此时的money就变成了 2，这就出现数据混乱

[^临时变量]: 在python内部中 一个 num +=1 的运算会分成3部分: tmp = num >> tmp = tmp + 1 > >num = tmp， 上面临时变量指的就是tmp，时间片就是在 num = tmp这一步用完的。

什么是锁

每一把锁对应一把钥匙，一旦某个变量，钥匙一旦被一个程序拿走，所有需要拿钥匙的程序都需要等待钥匙还回来了拿到钥匙才能运行

2个方法都加了锁，所以必然只有一个进程拿到了钥匙，另外一个进程没有拿到，另外一个进程就得等这钥匙被还回来

消费者生产者模式

为什么要用

并发进程中：负责生产数据的进程就被称之为：生产者，

​ 负责使用和消费数据的进程被称之为：消费者

程序的运行是要讲究尽量快的，在并发中，如果生产者生产数据过慢，消费者处理数据过快，消费者就要等待生产者，反之，如果生产者生产速度过快，消费者消费数据过慢，生产者就需要等待消费者。这样就照成了时间和资源的浪费，这种问题被称之为 强耦合， 所谓的 消费者生产者模式 就是为了这类问题提出的一个解决模式

生产者消费者模式

生产者消费者模式就是通过一个容器来解决生产者和消费者之间的强耦合问题，生产者和消费者之间不直接通信，而是通过一个阻塞队列通讯，所以生产者生产完数据后不需要等待消费者处理，直接放入阻塞队列，消费者和不在找生产者要数据，而是通过阻塞队列获取，阻塞队列就相当于是一个缓冲区，一定程度上平衡了生产者和消费者的处理能力

例如： 在全栈开发过程中，前端接收客户请求，后端处理逻辑请求，当某一时间服务端压力过大，无法立马处理客户的请求，就可以把客户的请求放在缓存队列中，后端处理完了一批请求后立马读取下一批的处理。

JoinableQueue 队列模块

JoinableQueue继承了Queue模块，且多了2个方法， .task_done()和.join()

JoinableQueue 说明
.task_done() 每获取队列中被 .get() 一个数据, 使用 .task_done() 就通知 .join()一下
.join() 阻塞，直到 .task_done()通知 队列中的所有项目都已获取并处理完毕[^详细]，就取消阻塞

[^详细]: join() 知道队列的长度，每当使用者进程 get()一个数据，然后 调用task_done()就会通知 .join() 拿走了一个数据，当 .join() 检索该项完成了所有相关工作。队列中所有的项目全部被获取完毕后，就取消阻塞，
