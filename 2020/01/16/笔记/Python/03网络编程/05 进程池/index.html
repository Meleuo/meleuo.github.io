进程池

在生产环境中，大多数情况是无法预料需要执行多少任务的，并且，不可能按照任务的数据来开启对应的进程，因为如果需要开启的进程数过多仅仅是开启进程这个操作就需要消费大佬的时间，且一个系统不仅要开启一个进程，开启后还要对他进行调度管理，单台计算器的资源是有限的，大量的进程会严重影响计算器的性能。

进程池：就是一个池子，里面有固定数量的进程[^固定数量]，这些进程一直处于待命状态，一旦有任务过来就会立马的去执行，且进程池还会自动帮你管理进程池中的进程，编写代码时能用进程池解决就不要用进程解决

[^固定数量]: 一般是CPU逻辑核数 + 1

使用进程池

  • p.map(func,iterable)
    • func对应要执行的任务函数，
    • iterable对应一个可迭代，并且 依次 传给任务函数执行
  • p.apply(func,args=())
    • func对应任务函数，args传入参数
    • 同步处理任务，进程池中的所有任务都是普通进程，主进程需要等待进程池中的所有任务执行结束，
    • 进程池每次返回的就是 任务函数的返回结果，
  • p.apply_async(func,args=(),callback=None)
    • func对应任务函数，args传入参数，
    • 异步处理任务，进程中的所以进程都是守护进程，因此主进程代码结束，进程池中的所有进程也就凉了，想要获取异步进程池的所有的返回对象需要在代码下面 添加 p.close() 和 p.join()
    • 程池每次返回的是一个对象要用 .get()获取对应任务函数的返回结果，
    • callback=None定义 回调函数，当进程池中的进程执行完后，返回的结果会以形参的形式传递给 callback 定义的回调函数，由回调函数进一步操作结果

从结果可见，即使使用了回调函数 进程池返回的对象还是方法函数的返回值，

效率对比

可见就单单启动进程，apply和apply_async就远快于map方法，所有map了解一下就好了~

为什么异步效率是远高于同步呢 ？

因为进程池中虽然有 os.cpu_count() +1个进程，但是启用的是同步模式，每个任务都需要等待上一个认为执行完毕了才会进入进程池被某个进程运行，也就是说同步模式下同一时间进程池中是只有一个进程在工作的

而异步则是 不管你上个任务有没有执行完，就立马加载下一个任务，所以异步模式下进程池中的每个进程都在工作，进程池中有5个进程，一下就处理5个任务，接下来哪个进程处理完任务了，就马上去接收下一个任务

当进程池中处理完任务的来返回的结果可以交给回调函数，有回调函数进一步进行处理

异步的回调函数

进程的任务函数的返回值，被当成回调函数的形参接收到，以此进行进一步的处理操作
回调函数是由主进程调用的，而不是子进程，子进程只负责把结果传递给回调函数
