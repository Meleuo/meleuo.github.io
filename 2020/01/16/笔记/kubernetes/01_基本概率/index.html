    Pod/Pod控制器

    Pod

    • Pod是K8s里最小的逻辑单元
    • 1个Pod里面可能有多个容器， 一个P里面有多个C，并且多个c中共享 UTS NET IPC名称空间
    • Pod是一个豌豆荚，容器相当于豌豆
    • 一个Pod里运行多个容器， 又叫 边车模式

    Pod控制器

    • Pod控制器是Pod启动的一种模板，用来保证K8s里启动的Pod安装预期的运行（副本数， 生命周期， 健康状态检查）
    • K8s提供众多的Pod控制器
      • Deployment
      • DaemonSet
      • ReplicaSet
      • StatefulSet
      • Job
      • CronJob

    Name /Namespace

    Name

    • 由于K8s内部，使用”资源” 来定义一种逻辑概念（功能）故每种 “资源” 都需要定义一个“名称”
    • “资源”有Api版本(ApiVersion) 类别（Kind）, 元数据（metadata）， 定义清单（spec）， 状态（status）等配置信息
    • “名称” 通常定义在 “资源” 的 “元数据” 信息里

    Namespace

    • K8s内部的虚拟集群组
    • 同组不可同名，不同组可以同同名
    • k8s 里默认存在的名称空间有： default， kube-system, kube-pulic

    Label/Label选择器

    Label

    • 标签是k8s特色的管理方式，便于分类管理资源对象
    • 一个标签可以对应多个资源， 一个资源也可以有多个标签，他们是多对多的关系
    • 一个资源拥有多个标签，可以实现不同维度的管理
    • 标签的组成： key=value
    • 和标签类似的还有一个 “注解” （annotations）

    Label选择器

    • 给资源打上标签后， 可以使用标签选择器过滤指定的标签。

    • 标签选择器目前有两个： 基于等值关系（等于，不等于）， 集合关系（属于，不属于， 存在）

    • 许多资源支持内嵌的选择器字段

      • matchlabels
      • matchExpressions

    Server / Ingress

    Service

    • 在K8s的世界里，虽然每个Pod都会被单独分配一个单独的IP地址，但是这个IP地址也随之Pod的生命周期而销毁和创建
    • Service(服务) 就是用来解决这个问题的核心概念
    • 一个Service可以看作一组提供相同服务Pod对外的访问接口
    • Service作用域那些Pod是通过标签选择器定义的

    Ingress

    • Ingress是K8s集群里工作在OSI为了模型参考模型下的，第七层的应用，对外暴露的接口
    • Service只能使用L4流量调度， 表现新手是ip + port
    • Ingress 则可以调度不同的业务域， 不同URL访问路径的业务流量
